Engine Info ⚙️
Data structures & algorithms powering FinanceOS
🔬 All features in FinanceOS are backed by real DSA concepts from your curriculum. This page explains what runs under the hood.
📦 Data Structure
Doubly Linked List — Transaction Ledger
Every transaction is a node in a doubly linked list. Previous/next pointers allow instant traversal for "undo last transaction", forward/backward navigation in history, and O(1) insertion at head/tail.
Insert: O(1) Traverse: O(n) Delete: O(1)
🔑 Data Structure
Hash Map — Instant Transaction Search
Transactions are stored in a hash map keyed by ID and description tokens. The search bar uses hash-based lookup for O(1) average-case search. Collision handling via open addressing (linear probing).
Search: O(1) avg Insert: O(1) avg Separate Chaining
⚡ Algorithm
Priority Queue (Max-Heap) — Budget Alerts
Budget overruns are stored in a max-heap priority queue sorted by overspend severity. The most critical alerts bubble up first. Each budget check runs in O(log n) heapify time, ensuring alerts are always prioritized correctly.
Enqueue: O(log n) Peek Max: O(1) Heapify: O(log n)
🔃 Algorithm
Merge Sort — Transaction Sorting
The "Sort by Amount" and "Sort by Date" features use Merge Sort for guaranteed O(n log n) performance on any input. Merge sort is ideal here as it's stable — transactions with same amount preserve their relative date order.
Best: O(n log n) Worst: O(n log n) Stable: Yes
🌳 Data Structure
BST — Category Budget Tree
Budget categories are organized in a Binary Search Tree ordered by spending amount. Finding "all categories over budget" is an efficient in-order traversal. Inserting a new budget category takes O(log n) on average.
Insert: O(log n) Search: O(log n) In-order: O(n)
🔍 Algorithm
Binary Search — Date Range Filter
Transactions sorted by date use Binary Search to find the boundary positions for any date range query in O(log n). The filter "Show transactions between Jan–Mar" uses two binary search calls to find start and end indices.
Range Find: O(log n) Requires: Sorted Space: O(1)
🔃 Algorithm
Quick Sort — Monthly Aggregation
Monthly income/expense summaries are produced by partitioning and sorting using Quick Sort with a randomized pivot strategy. Average case O(n log n) with O(log n) stack space. Used for chart data preparation.
Avg: O(n log n) Worst: O(n²) In-place: Yes
📚 Data Structure
Stack + Queue — Undo/History
Transaction undo uses a Stack (LIFO) — pop the last transaction to undo. The "recent transactions" view uses a Circular Queue that maintains the 10 most recent entries efficiently. Queue for order, Stack for undo.
Stack Push/Pop: O(1) Queue En/Dequeue: O(1) Circular: O(1)
📋 Data Structure
Polynomial Evaluation — Infix to Postfix
The financial calculator internally converts infix expressions (3+4×2) to Postfix (RPN) using a stack-based Shunting-Yard algorithm. Postfix evaluation uses another stack — the same technique used in all modern calculators.
Conversion: O(n) Evaluation: O(n) Uses: Stack ADT
🔢 Algorithm
Hashing — Extendible + Separate Chaining
Category spending totals use a hash table with separate chaining for collision resolution. Extendible hashing allows the table to dynamically resize when load factor exceeds 0.75, maintaining O(1) amortized performance.
Lookup: O(1) Load Factor: 0.75 Resize: O(n)
🔢 Algorithm
Bubble/Selection/Insertion Sort
Small lists (≤ 20 transactions) use Insertion Sort as it's optimal for nearly-sorted or small arrays. The sort-by feature auto-selects between insertion sort (small) and merge sort (large) based on list size — adaptive sorting strategy.
Best: O(n) Avg: O(n²) Small n: Fastest
📊 Algorithm
Circular Queue — Salary Scheduler
The salary auto-credit scheduler uses a circular queue to track monthly credit events. On the 1st of each month, the queue dequeues the salary event and enqueues the next month's event — continuous salary tracking with O(1) operations.
Enqueue: O(1) Dequeue: O(1) Space: O(1)
